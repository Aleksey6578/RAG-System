# 1. Как исправлять (приоритетный план)

## 🔴 Шаг 1. Убрать уничтожение данных (chunking)

### Было

* жёсткие лимиты на секции
* потеря ~40% данных

### Делать

**Заменить hard limits → soft sampling**

#### Вариант A (простой)

```text
max_chunks_per_section_type → убрать
```

#### Вариант B (правильный)

Балансировка:

```text
если section > N:
    брать:
        - первые k
        - последние k
        - случайные k
```

#### Вариант C (лучший для ВКР)

**Weighted sampling**

```text
P(chunk) = importance(section_type)
```

пример:

| section      | weight |
| ------------ | ------ |
| competencies | 3.0    |
| outcomes     | 2.0    |
| content      | 1.0    |
| assessment   | 1.2    |

---

## 🔴 Шаг 2. Переделать chunking (ключевая проблема)

Сейчас:
**fixed chunking (по токенам)** → мусор

### Нужно: Semantic chunking

#### Метод:

1. Разбивка по структуре документа:

   * заголовки
   * списки
   * таблицы

2. Затем:

```text
chunk = 200–400 tokens
overlap = 50–80
```

3. Главное:

* **не смешивать разные секции в один chunk**

---

## 🔴 Шаг 3. Убрать тупую дедупликацию

### Было

* удаление 30% чанков

### Делать

#### Вместо text-dedup → embedding-dedup

```text

    удалить
```

* учитывать metadata:

```text
(section_type, source_doc)
```

## 1. Ключевые проблемы RAG (по факту анализа корпуса + логов)

### 1.1. Критичные (влияют на retrieval напрямую)

**1. OCR не работает → провал покрытия**

* Факт: *Хайкин = 0 чанков* 
* Следствие:

  * отсутствуют embeddings для базового источника по DL
  * retrieval по запросам типа *“перцептрон / backpropagation”* деградирует

**Что исправить (в архитектуре):**

* В отчёте явно прописать fallback pipeline:

  ```
  PDF → (if text layer empty) → OCR → text → chunking → embeddings
  ```
* Добавить метрику:

  * `coverage_per_source = chunks_count > 0`
* Ввести контроль:

  * % документов с 0 чанков (target: 0%)

---

**2. Tokenizer fallback (len×0.375)**

* Факт: используется эвристика вместо real tokenizer 
* Ошибка:

  * нарушается semantic chunking
  * разрываются смысловые блоки

**Что исправить:**

* В отчёте:

  * указать, что используется **token-based chunking**
  * модель: `bge-m3 tokenizer`
* Формализация:

  ```
  chunk_size = 512 tokens
  overlap = 50–100 tokens
  ```
* Добавить сравнение:

  * heuristic vs tokenizer (ошибка 10–25%)

## 2. Проблемы корпуса (corpus_analysis.json)

### 2.1. Перекос доменов

**Сильный перекос в AI:**

* AI/ML: 15 дисциплин
* низкоуровневые/сети/БД: 1–2 

**Следствие для RAG:**

* retrieval biased:

  * запрос → попадает в AI даже если не должен
* embedding space “перетягивается”

---

### 2.2. Дубли и шум

Примеры:

* "Безопасность информационных систем" — 2 раза 
* "Системное программирование" в двух доменах (Прочее + системное)

**Проблема:**

* ухудшается clustering
* retrieval может возвращать дубликаты

**Что добавить в отчёт:**

* этап **deduplication**:

  ```
  sim(doc_i, doc_j) > 0.9 → merge
  ```
* нормализация названий дисциплин

---

### 2.3. Неправильная классификация доменов

Пример:

* "Компьютерная графика" → Математика / Алгоритмы 

**Ошибка:**

* домен используется как фильтр retrieval
* искажается routing запроса

**Что исправить:**

* в отчёте добавить:

  * этап **domain classification (classifier / rules)**
  * или manual mapping

---

### 2.4. Недостаточное покрытие (ключевая проблема RAG)

Критичные домены (<3):

* Системное ПО
* Сети
* БД
* IoT
* Графика 

**Следствие:**

* RAG не может:

  * делать grounding
  * генерировать РПД в этих областях

**Важно:**
Ты уже это исправляешь (синтетические РПД) — это правильно.

# **Анализ и замечания по RAG-модулю**

## **1. Общая характеристика состояния системы**

Текущая реализация RAG-модуля демонстрирует формальную работоспособность: генерация результатов происходит, метрики не являются нулевыми. Однако детальный анализ логов, корпуса и метрик показывает системные проблемы на уровне архитектуры.

Ключевая особенность:

> значительная часть информации теряется **до этапа retrieval**, что приводит к снижению качества генерации.

---

# **2. Критические проблемы (влияние на retrieval и качество генерации)**

## **2.1. Потери данных при chunking**

### Факты:

* исходные записи: **4409**
* после chunking: **2536**
* потери: **≈42%**

### Причина:

жёсткие ограничения:

```
max_chunks_per_section_type
assessment = 12
```

### Последствия:

* снижение **Recall**
* неполное покрытие корпуса
* генерация с опорой на недостаточные данные

---

## **2.2. Некорректный chunking (отсутствие семантики)**

### Текущее состояние:

* разбиение по токенам (или эвристике len × 0.375)
* средний размер: ~173 токена при max=400

### Проблема:

* разрыв смысловых блоков
* смешивание разных секций

### Последствия:

* низкое качество embeddings
* слабые similarity-score

---

## **2.3. Агрессивная дедупликация**

### Факты:

* удалено: **~30% чанков (1017)**

### Проблема:

* удаление по тексту без учета контекста
* игнорирование metadata:

  * section_type
  * source_doc

### Последствия:

* потеря вариативности формулировок
* ухудшение diversity retrieval

---

## **2.4. Низкое качество similarity-score**

### Факты:

* диапазон: **0.45–0.66**
* норма: **0.7–0.85**

### Причины:

1. плохой chunking
2. шум в тексте
3. неэффективные embeddings

### Последствия:

* слабый сигнал релевантности
* увеличение галлюцинаций

---

## **2.5. Отсутствие reranking**

### Текущая схема:

```
query → top-k (vector search)
```

### Проблема:

* отсутствует второй этап отбора

### Последствия:

* релевантные чанки могут не попадать в top-k
* ухудшение итогового контекста

---

## **2.6. Отсутствие OCR**

### Факт:

* часть документов: **0 чанков**
* причина: `tesseract is not installed`

### Последствия:

* потеря источников знаний
* снижение coverage корпуса

---

## **2.7. Ошибки embedding слоя**

### Факт:

```
'NoneType' object is not subscriptable
```

### Проблема:

* отсутствие retry/fallback механизма

### Последствия:

* потенциальная потеря данных
* нестабильность пайплайна

---

### Последствия:

* ошибки в domain filtering
* нерелевантный retrieval

# **7. Архитектурные недостатки**

## **Текущий pipeline**

```
Raw → Clean → Chunk (hard limits) → Dedup → Embed → Index
```

## **Проблема**

потери информации происходят **до retrieval**

---

## **Рекомендуемый pipeline**

```
Raw
→ OCR (при необходимости)
→ Cleaning
→ Semantic chunking
→ Light dedup (embedding-based)
→ Балансировка данных
→ Embedding
→ Retrieval (top-k)
→ Reranking
```

## **Финальный вывод**

Текущая реализация RAG-модуля демонстрирует базовую работоспособность, однако качество генерации ограничено архитектурными ошибками на этапах preprocessing и retrieval. Основным узким местом системы является не генерация, а недостаточная полнота и релевантность извлекаемого контекста. Исправление этих компонентов является ключевым условием повышения качества всей системы.
